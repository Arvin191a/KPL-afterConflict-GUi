This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

GUI-KPL is a Windows Forms (.NET 8) inventory-management desktop app built as a software-engineering coursework project (KPL = *Konstruksi Perangkat Lunak*). The UI language and most identifiers/comments are Indonesian (`Barang` = item, `Akun` = account, `Cari` = search, `Tabel` = table). Data is persisted to local JSON files — there is no database.

## Build & Run

```powershell
dotnet build GUI-KPL.sln            # build whole solution
dotnet run --project View           # launch the WinForms app (entry point)
```

`View` is the startup project; its entry point is `View/Program.cs`, which opens the `Login` form. `Main` is a class library (despite `OutputType=Exe`, its `Progam.cs` `Main` is a vestigial stub and is not the app entry point). There are no tests and no lint configuration in this repo.

Building must happen on Windows — `View` targets `net8.0-windows` with `UseWindowsForms=true`.

## Architecture

Two projects participate in the solution; `View` references `Main`:

- **`Main`** — domain models only, namespace `Main.Model`:
  - `Akun` (account: Username, Nama, Password, Role).
  - `Barang` (item: NamaBarang, KodeBarang, HargaBarang, JumlahBarang, Status). Stock changes go through `Barang.UpdateStok`, which drives a **state automaton**: `Habis` (0), `BatasKritis` (1–5), `Tersedia` (>5). `JumlahBarang` and `Status` have private setters, so stock must be mutated via `UpdateStok`, never directly. `Status` serializes to its integer enum value in JSON.
  - `Barang.Add` / `Barang.RemoveAt` are `NotImplementedException` stubs — do not call them.
- **`View`** — all WinForms screens. Each screen is a `Form` paired with a `*.Designer.cs`. Navigation is form-to-form: `Login` → `Homepage` (Admin) or `HomepageUser` (User); `Homepage` opens `CreateBarang` / `CariBarang` / `DeleteBarang` / `TabelBarang`.

### Role-based routing (table-driven)

`Login` authenticates against `dataAkun.json` and routes by `Role` using a `Dictionary<string, Func<Form>>` (`_roleRoutingTable`): `"Admin"` → `Homepage`, `"User"` → `HomepageUser`. `Register` similarly uses a table-driven `ValidationStatus` enum → message dictionary pattern. When adding a role or validation case, extend these dictionaries rather than adding `if`/`else` branches — the table-driven style is a deliberate course requirement.

### Data persistence

Persistence is done ad hoc: most forms each define their own `ReadJSON`/`WriteJSON` using `System.Text.Json` (`JsonSerializer`). There is **no shared repository/service layer** — the same read/write logic is duplicated across `Login`, `Register`, `CreateBarang`, `CariBarang`, `DeleteBarang`, etc.

Most forms hardcode absolute paths:

```
C:\KPLProj\Main\Data\dataBarang.json
C:\KPLProj\Main\Data\dataAkun.json
```

This ties the app to the `C:\KPLProj` checkout location. `TabelBarang` is the exception — it uses `Path.Combine(AppDomain.CurrentDomain.BaseDirectory, "Data", "dataBarang.json")`, reading the copy that `View.csproj` copies into the build output (`<None Include="..\Main\Data\dataBarang.json">` with `CopyToOutputDirectory=PreserveNewest`). As a result, **the two paths can diverge**: read/write forms hit the source `Main/Data` file while `TabelBarang` reads the output-dir copy. Keep this in mind when data appears stale, and prefer the relative `BaseDirectory` approach for any new file access. Note `dataAkun.json` is **not** copied to output.

## Conventions & Gotchas

- Indonesian naming throughout; keep new identifiers consistent with surrounding code (e.g. `dataBarang`, `namaBarang`).
- Passwords are stored and compared in plaintext in `dataAkun.json` (course project — do not treat as production auth).
- `homepage_gui/` is a standalone scaffold project **not** included in `GUI-KPL.sln` and not part of the app; ignore it unless explicitly working on it.
- Designer files (`*.Designer.cs`) are auto-generated by the Visual Studio WinForms designer; edit layout through the designer when possible rather than hand-editing.
- `bin/`, `obj/`, and `.vs/` are checked into git in this repo — avoid committing churn in those directories when making source changes.
